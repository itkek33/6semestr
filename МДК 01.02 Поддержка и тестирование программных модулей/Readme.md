**Требывания к тестировщику:**

1. Теория тестирования и документация
2. Знания предмета, который тестируется
3. Использовать инструметы автоматического тестирования

**Тестировавние:**

1. Включен во все активности жизненного цикла
2. Статичекая - без исполнения кода программного продукта, динамическая - во время выполнения программного обеспечения
3. Оцениватеся ожидание и рпеальность

**Объект тестирования** - это программа (что сравнивается) 
- отдельный модуль
- компонет (несколько модулей
- подсистема
- система

**Базис тестирования** - это документация (с чем сравниватеся) 

**Дефект (баг):** 
1. Полшрамма не делает, что должна
2. Программа делает то, что не должна
3. Программа делает то, что не упоминалось
4. Программа неудобна

2. Ошибка в проектировании: неправильно разработаная база данных
3. Ошибка в разработке: синтаксическае ошибки кода

**Обеспечение качества QA** - разработчик

**Конотроль качеста QC** - тестировщик 
1. Тестирование

**7 принципов тестирования**
1. Наличие дивекта (все найти невозможно)
2. Исчерпывающее тестирование недостежимо
3. Чем раньше начать тестирование, тем лучше
4. Скопление дефектов
5. Парадокс пестицида (повтаряя одно и тоже новые дефекты не найти)
6. Нужен контекст
7. Не забываем о пользователе

# ЛК2 12.02.2024 TDD. Разработка через тестирование.

**Жизненный цикл ПО**

Модели жизненного цикла:
1. Каскадная модель:
   - Под тестирование выделен только один этап, дефект во время эксплуатации нельзя исправить. 
   - Нет плана тестирование
2. V модель:
   - Тестируется каждый этап, но возврата по прежнему нет
3. Итеративная или инкрементальная модель:
![image](https://github.com/FominIgor/6-semestr/assets/113089549/b9762348-ebfc-4eb2-969e-1022e864b7df)
   - Большие объемы реграссионного тестирования
   - Отсуствие плана 
4. Спиральная модель:  
   - Сложно тестировать на ранних этапах, но тестирование становится ключевым этапом.

Уровни тестирования:
1. Компонентное
2. Интеграционное
3. Системное
4. Приёмочное

Тестирование бывает статическим и динамическим
1. Статическое - без исполнения кода
2. Динамическое - с исполнением кода

Тесты гугл карты 
1. Функциональные тесты:
   - Проверка функциональности входа в систему с использованием правильных учетных данных.
   - Тестирование функции вывода маршрутов.
   - Проверка правильности отображения данных на странице профиля пользователя.

2. Тесты производительности:
   - Измерение времени загрузки веб-страницы при различных уровнях нагрузки.
   - Оценка времени ответа сервера при обработке большого количества запросов.
   - Тестирование производительности базы данных при выполнении сложных запросов.

3. Нагрузочные тесты:
   - Симуляция одновременного доступа к системе с большим количеством пользователей.
   - Измерение максимальной нагрузки, которую может выдержать система без снижения производительности.
   - Тестирование на устойчивость при длительной нагрузке на систему.

4. Тестирование совместимости:
   - Проверка работы веб-приложения на различных браузерах (Chrome, Firefox, Safari, Edge).
   - Тестирование совместимости мобильного приложения с разными версиями операционных систем (iOS, Android).
   - Проверка работы приложения на разных разрешениях экранов и устройствах.

1. Позитивные тесты:
   - Ввод корректных данных в форму регистрации и проверка успешной регистрации пользователя.
   - Проверка корректной обработки правильного ввода данных в поле поиска.
   - Проверка правильной работы функциональности добавления нового элемента в список.

2. Негативные тесты:
   - Попытка входа в систему с неверным паролем и ожидание сообщения об ошибке.
   - Ввод некорректного формата электронной почты при регистрации и проверка обработки ошибки.
   - Попытка удалить маршрут.

3. Исследовательские тесты:
   - Изучение внутренней структуры приложения для выявления потенциальных уязвимостей.
   - Поиск неожиданного поведения приложения при вводе различных комбинаций данных.
   - Тестирование функциональности, не предусмотренной в документации, для проверки скрытых возможностей или ошибок.

1. Модульное тестирование:
   - Тестирование функции вывода маршрута 
   - Тестирование метода вывода карты 
   - Тестирование класса для работы с базой данных пользователей 

2. Интеграционное тестирование:
   - Тестирование взаимодействия между клиентской и серверной частями приложения
   - Тестирование интеграции различных модулей системы управления
   - Тестирование взаимодействия между различными сервисами веб-приложения

3. Системное тестирование:
   - Тестирование работы всего приложения
   - Тестирование производительности системы при большой нагрузке
   - Тестирование совместимости приложения с различными операционными системами

# ЛК3 19.02.2024 Дефекты.

**Отчет о дефекте** - это документ, который содержит информацию о том, что может привести систему к невозможности выполнить требуемую функцию.
![image](https://github.com/FominIgor/6-semestr/assets/113089549/255b3d3b-5e46-46f6-9aab-ee97e63a3cfb)

Отчет о дефекте включает в себя:
1. Идентификатор
2. Краткое описание: характер проблемы, границы дифекта, последствия дифекта. 
3. Подробное описание: ожидаемые и фактические результаты. 
4. Влияние - степинь дефект. Критичность и приоритет.

**Причины почему дефекты не устраняют**
1. Нехватка времини
2. Дефект вовсе не дефект
3. Устранить неисправность слишком рисковано
4. Это того не стоит
5. Неэффективный отчет о дефектах

Упражнение:
1. Идентификатор: 1
2. Краткое описание: Сложение четных чисел происходит неверно
3. Подробное описание: В ходе сложения двух четных результат больше на 1, чем должно было быть
4. Шаги:
   
   Открыть калькулятор
   
   Нажатие на четное число №1
   
   Нажатие на +
   
   Нажатие на четное число №2
   
   Нажатие на =
   
6. Ожидаемый результат: четное число 1 + четное число 2 = четная сумма чисел
7. Фактический результат: четное число 1 + четное число 2 = нечетная сумма чисел
8. Влияние: критический дефект, сложение важная математическа операция.
